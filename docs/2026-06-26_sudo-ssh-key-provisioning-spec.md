# Sudo SSH key provisioning — spec

Raised & validated 2026-06-26. Give Spinup **privileged write access** to a server
via a SpinupWP **sudo user**, so it can do things the SpinupWP API cannot — starting
with: drop an SSH key into a site user's `authorized_keys`. This is the first time
Spinup crosses from read-only SSH into **write-over-SSH**, and it's the shared
primitive under backlog item 5 (creation + SSH clone) and item 9 (Bedrock
composer→push→deploy).

Builds on: the read-only SSH transport in `src/lib/ssh.ts` (`Bun.spawn(["ssh", …])`
with `BatchMode=yes` + `ControlMaster` persistence), the write scaffolding from PHP
upgrade (`mutate()` is API-only, but the **confirm-before-firing overlay** in
`src/ui/views/PhpUpgrade.tsx` is the pattern to reuse), and config persistence in
`src/config.ts` (chmod-600 `~/.config/spinupwp-tui/config.json`).

## The product

> "I can create sudo users on a server (key login + a sudo password). Let Spinup use
> that access so that when we create sites, it can add my SSH key to all of them in
> one pass — without me checking boxes in the SpinupWP UI for each one."

## Why this can't go through the API (probed 2026-06-26)

The SpinupWP API has **no** SSH-key or sudo-user management surface. Unauthenticated
route probes:

- `/v1/servers`, `/v1/sites`, `/v1/events` → `"Unauthenticated."` (routes exist)
- `/v1/ssh-keys`, `/v1/ssh_keys`, `/v1/sudo-users`, `/v1/sites/{id}/ssh-keys`,
  `/v1/servers/{id}/ssh-keys`, `/v1/sites/{id}/users` → **`"route … could not be
  found."`** (don't exist)
- Curiosity: `/v1/ssh-key` (singular) auth-gates but has no `/{id}` and no plural —
  likely a POST-only account-key utility, **not** per-user assignment. Worth one
  authenticated peek someday; does not change this plan.

So writing `authorized_keys` directly over SSH is the only path. Reaching *other*
site users' home dirs requires root → **sudo**.

## What the SpinupWP UI actually does (context)

In SpinupWP, SSH keys are saved at the **account** level; the per-site-user and
per-sudo-user pages just **associate** (checkbox) account keys with a user, and
SpinupWP renders that into `/sites/<domain>/.ssh/authorized_keys` (mode 600, owned
`siteuser:siteuser`). The API can't toggle those checkboxes; our sudo append
replicates the *effect* in bulk.

## Clobber test — VERIFIED SAFE (2026-06-26)

The one real risk was: does SpinupWP rewrite `authorized_keys` wholesale on reconcile
and wipe a key we appended? Tested on a throwaway key against
`spinup02@spinup.02.wenmarkdigital.com` (full cleanup after):

- Adding/removing account keys in the UI **merges** into the file — no wipe-and-rewrite.
- Removing **all** selected account keys stripped SpinupWP's managed keys but **left
  the foreign (non-account) key untouched.**
- SpinupWP keys off the **exact key string**, not file position.

**Conclusion:** an appended key survives reconciliation — *as long as it isn't a string
SpinupWP also manages as a saved account key* (un-checking that account key in the UI
would remove the matching line). → Use a **dedicated key NOT saved in the SpinupWP
account** for a truly untouchable machine identity. (See memory:
`spinupwp-authorized-keys-reconcile`.)

## Decisions (user, 2026-06-26)

1. **Dedicated machine key, not the personal account key.** Generated by Spinup,
   never added to the SpinupWP account, so SpinupWP never manages/removes it.
   - **Label it** via the public-key comment so its origin is obvious wherever the
     line is seen: comment = `spinup-tui@<local-hostname>` (e.g.
     `…AAAAC3Nz… spinup-tui@Michaels-MacBook-Pro.local`). Open to
     `spinup-tui-<accountSlug>` if we want account-scoping instead of host-scoping.
   - **Storage:** generated lazily on first use into the config dir —
     `~/.config/spinupwp-tui/keys/spinup-tui` (+ `.pub`), private key chmod 600.
     ed25519, no passphrase (it's an unattended machine identity).
2. **Single-site first.** v1 = "add the Spinup key to *this* site's user." The bulk
   "add to every site user on the server in one pass" is a follow-up layered on the
   same primitive.
3. **Sudo password: in-memory only** for v1. Prompt once per session (when the first
   privileged action fires), hold it for the run, never persist. macOS Keychain
   persistence is a deferred opt-in.

### `s ssh` interactive connect — NOT modified (decided)

`openSshSession()` (`src/lib/open.ts:56,63`) opens a plain `ssh user@host` in the
user's terminal, using **their** personal key/agent. Keep it that way:

- The `spinup-tui` key is a **machine identity** for Spinup's *automated* work; a
  human opening a shell wants to be **themselves** (agent forwarding, `git push` as
  them, their own sudo). Swapping in the machine key would strip that.
- The "just works even where my personal key was never added" upside is real, so spec
  it as a **future opt-in fallback** for `s ssh` (`ssh -i <spinup-tui> -o
  IdentitiesOnly=yes …` when the user's default identity isn't present), **not** a
  replacement. Out of scope for v1.

## Transport & sudo mechanics

1. **Login:** key-based as the **sudo user** (e.g. `spinup02@<ip>`), via the existing
   `ssh.ts` `BatchMode=yes` + `ControlMaster` transport. No password at this step.
   (For v1 the sudo user logs in with the *personal* key that already works; the
   `spinup-tui` key is what gets *deployed*, not necessarily what logs in. Revisit
   if/when we consolidate Spinup's own login identity onto the machine key.)
2. **Escalate, non-interactively — do NOT drive an interactive `sudo -i` shell.**
   Run one shot: `ssh … <sudouser>@<ip> "sudo -S -p '' bash -s"`, feeding
   `password\n` then the script body on **stdin** (flip `ssh.ts`'s current
   `stdin:"ignore"` → `stdin:"pipe"` on the privileged path). `-S` reads the password
   from stdin; `-p ''` silences the prompt. **Password never appears in argv** (so not
   in remote `ps`/history). Modern Ubuntu (SpinupWP's OS) has no `requiretty`, so this
   works over a non-TTY ssh.
3. **Probe first:** `sudo -n true` to detect "can't sudo" / "no password needed" with
   a clean error *before* sending the secret.
4. **Act — one idempotent script** (batch the whole op into a single `bash -s`, like
   `REMOTE_SCRIPT` batches reads):
   ```sh
   U="<site_user>"; H="/sites/<domain>"; K="<spinup-tui pubkey line>"
   install -d -m 700 -o "$U" -g "$U" "$H/.ssh"
   touch "$H/.ssh/authorized_keys"
   grep -qxF "$K" "$H/.ssh/authorized_keys" || printf '%s\n' "$K" >> "$H/.ssh/authorized_keys"
   chown "$U:$U" "$H/.ssh/authorized_keys"; chmod 600 "$H/.ssh/authorized_keys"
   ```
   `grep -qxF` = idempotent (re-running never duplicates). Verify by reading back.

## Per-server config (new)

Sudo user + (in-memory) password are **per server**. The current global `sshUser`
override is one value; add a per-server map keyed by `server.id` (same pattern as
`localSites`/`jobs` in `src/config.ts`):

```ts
sudoUsers: Record<string /* serverId */, { user: string }>  // username only; password stays in-memory
```

`Server` already carries `ip_address` and `ssh_port`. The sudo *password* is held in
the runtime store (e.g. `Map<serverId, string>`), prompted on first use, never written
to disk in v1.

## Guardrails (reuse existing posture)

- **Confirm-before-firing overlay** (the `PhpUpgrade.tsx` pattern). This is the most
  powerful write Spinup has — it must be explicit.
- **Show the exact remote command** before firing (transparency; fits "name by
  outcome, teach in context").
- Respect memory `feedback-confirm-before-prod-writes`: never fire a privileged write
  without explicit user go-ahead.
- Never log the password; never put it in argv; in-memory only.

## v1 surface (single-site)

New helper in `src/lib/ssh.ts`, returning the same shape as `fetchServerHealth`:

```ts
// Append Spinup's dedicated machine key to a site user's authorized_keys, via sudo.
grantSiteSshKey(server: Server, site: Site, opts: { sudoUser: string; sudoPassword: string })
  : Promise<{ ok: true; target: string } | { ok: false; target: string; error: string }>
```

Plus:
- `ensureSpinupKey(): Promise<{ path: string; pub: string }>` — lazily generate/return
  the dedicated keypair in the config dir.
- A UI action on a site (single key, behind the confirm overlay) that prompts for the
  sudo password (if not already held this session) and calls `grantSiteSshKey`.
